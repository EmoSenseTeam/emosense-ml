## 1. Общее описание проекта

### Что это за сервис

**EmoSense Student ML API** — это отдельный HTTP-сервис на **FastAPI**, который отвечает за **анализ данных**, связанных с эмоциональным состоянием и поведением студента. В текущей версии (MVP) реально реализован **модуль поведенческого анализа (behavioral)**: по числовым признакам (сон, экран, шаги, общение) считается **индекс `emotional_index`** и связанные с ним выводы.

Остальные модули (**разговорная эмоция**, **дневная агрегация**, **глубокий анализ за период**) в коде представлены **схемами данных и заготовками сервисов**, чтобы заранее зафиксировать **контракт API** и место для будущих ML-моделей.

### Зачем он нужен

Мобильное приложение или основной backend обычно не должны «таскать» с собой тяжёлую логику выводов по тексту, временным рядам и моделям. Им удобнее вызывать **один стабильный HTTP-endpoint**, получать **предсказание или индекс в JSON** и дальше решать, что показать пользователю и как сохранить в БД.

### Роль в системе EmoSense Student

В типичной схеме:

- клиент (мобильное приложение) общается с **основным backend** (пользователи, сессии, хранение данных);
- когда нужен **анализ** (behavioral, текст, тренды), основной backend (или отдельный оркестратор) отправляет запрос в **ML API**;
- ML API **валидирует вход**, **считает результат** и **возвращает структурированный ответ**.

Так **разделяют ответственность**: backend отвечает за продукт и данные, ML API — за вычисления и формат аналитики.

### Почему это отдельный сервис, а не часть mobile app

1. **Модели и правила меняются чаще**, чем релизы приложения в сторах. Отдельный сервис можно обновить на сервере без переустановки приложения у тысяч пользователей.
2. **Единая точка правды**: все клиенты (iOS, Android, веб) получают одинаковую логику scoring.
3. **Ресурсы**: тяжёлые модели (в будущем) лучше крутить там, где есть CPU/GPU и контролируемое окружение, а не на телефоне каждого студента.
4. **Безопасность и контракты**: вход и выход описаны схемами (Pydantic); проще тестировать, логировать и версионировать API (`/api/v1/...`).

## 2. Архитектура проекта (папки и файлы)

Ниже — логика типичного **layered** (слоистого) backend: **HTTP-слой** отделён от **бизнес-логики** и от **описания данных**.

```text
app/
  main.py              # точка входа приложения, подключение роутеров, обработчики ошибок
  api/v1/              # только маршруты HTTP (routers)
  schemas/             # контракты данных: запросы и ответы (Pydantic)
  services/            # бизнес-логика и (в будущем) вызовы моделей
  core/                # настройки окружения (config)
  utils/               # мелкие переиспользуемые функции
```

### Зачем так делить, а не писать всё в одном файле

Если положить маршруты, расчёты, настройки и схемы в один `mega.py`:

- файл быстро превращается в «кашу»;
- сложно тестировать логику отдельно от HTTP;
- сложно заменить baseline на ML, не задев HTTP;
- новому человеку непонятно, **где искать что**.

Разделение по папкам — это **соглашение команды**: каждый слой делает одну вещь.

### `app/main.py`

**Назначение:** создать объект приложения FastAPI (`app`), подключить роутеры с префиксами, задать глобальные обработчики ошибок и служебные маршруты (`/health`, `/`).

**Что здесь должно быть:** минимум логики; в идеале — сборка приложения из кусочков, а не «вся программа в одном месте».

### `app/api/v1/`

**Назначение:** описать **какие URL существуют**, **какой HTTP-метод**, **какая схема запроса/ответа**. В этом слое вызываются функции из `services/`.

**Что здесь должно лежать:** файлы с `APIRouter`, декораторы `@router.get/post/...`, при необходимости зависимости FastAPI (`Depends`).

**Чего здесь не должно быть:** длинных формул scoring, обращения к файлам моделей, сложной математики — это уезжает в `services/`.

В проекте есть, например:

- `behavioral.py` — рабочий `POST .../behavioral`;
- `conversation_emotion.py`, `daily.py`, `deep.py` — каркас под будущие endpoints.

### `app/schemas/`

**Назначение:** **контракт данных**. Это «договор» между клиентом и сервером: какие поля приходят, какие типы, какие ограничения (`ge`, `min_length`), как выглядит ответ.

**Почему Pydantic:** FastAPI глубоко интегрирован с Pydantic v2: из моделей автоматически строится **валидация**, **OpenAPI-документация** на `/docs`, **подсказки типов** в редакторе.

Файлы:

- `behavioral.py`, `conversation.py`, `daily.py`, `deep.py` — по одному домену;
- `__init__.py` — удобный реэкспорт моделей.

### `app/services/`

**Назначение:** **что именно считаем** и **как**. Здесь baseline-правила для behavioral; позже — загрузка весов, токенизация текста, инференс нейросети.

**Почему это правильно архитектурно:** HTTP-слой остаётся тонким. Вы можете поменять реализацию (правила → ML), не меняя URL и схемы — или менять схемы осознанно отдельным шагом.

### `app/core/`

**Назначение:** **настройки**, которые не хочется размазывать по файлам: имя приложения, версия, префикс API, флаги debug. У вас используется `pydantic-settings` с префиксом окружения `EMOSENSE_`.

### `app/utils/`

**Назначение:** мелкие функции без «бизнес-смысла домена», которые переиспользуются — например, **ограничение числа диапазоном** (`clamp_int` в `helpers.py`).

**Зачем отдельно:** чтобы не копировать `max(min(...))` в пяти сервисах и не ошибиться в границах.

### Корень репозитория

- `requirements.txt` — зависимости Python;
- `README.md` — краткая инструкция по запуску;
- **этот файл `READMEAbout.md`** — развёрнутое объяснение.

## 3. Полный разбор потока запроса `POST /api/v1/analysis/behavioral`

Рассмотрим **поток данных** от клиента до ответа — как цепочку шагов.

### Исходный HTTP-запрос

```http
POST /api/v1/analysis/behavioral HTTP/1.1
Host: localhost:8000
Content-Type: application/json

{
  "user_id": "student-42",
  "screen_time_hours": 9.5,
  "sleep_hours": 5.0,
  "physical_activity_steps": 3200,
  "communication_frequency": 2
}
```

### Шаг 1. Кто отправляет запрос

Отправитель в реальной системе чаще всего **не мобильное приложение напрямую**, а **основной backend** EmoSense (или тот же backend от имени клиента). Для отладки запрос может отправить **Swagger UI** (`/docs`), **Postman**, **curl** или тест.

### Шаг 2. Куда запрос попадает сначала

Запрос приходит на **ASGI-сервер** (например, **uvicorn**). Uvicorn передаёт его в приложение FastAPI — объект **`app`**, созданный в `main.py`.

### Шаг 3. Как `main.py` подключает роут

В `main.py` роутер behavioral подключается так (упрощённо по смыслу):

```python
app.include_router(behavioral_router, prefix=f"{settings.api_v1_prefix}/analysis")
```

`settings.api_v1_prefix` по умолчанию равен `"/api/v1"`. Значит префикс для этого роутера: **`/api/v1/analysis`**.

В файле `app/api/v1/behavioral.py` маршрут объявлен как `@router.post("/behavioral", ...)`.

**Итоговый путь:** `/api/v1/analysis` + `/behavioral` = **`/api/v1/analysis/behavioral`**.

### Шаг 4. Как FastAPI выбирает endpoint

FastAPI хранит **таблицу маршрутов**: метод (`POST`) + полный путь. Совпадение найдено — вызывается функция **`post_behavioral_analysis`**.

### Шаг 5. Как JSON превращается в Python-объект

Тело запроса — **JSON**. FastAPI читает байты тела и пытается разобрать JSON в структуру Python (словарь).

Дальше срабатывает объявление параметра:

```python
def post_behavioral_analysis(body: BehavioralAnalysisRequest) -> BehavioralAnalysisResponse:
```

Раз параметр аннотирован моделью Pydantic, FastAPI **передаст данные в конструктор** `BehavioralAnalysisRequest`.

### Шаг 6. Как срабатывает Pydantic

`BehavioralAnalysisRequest` наследуется от `BaseModel`. Pydantic:

- проверяет **наличие полей**;
- приводит типы, где это возможно;
- проверяет **ограничения** из `Field(...)` (например, `sleep_hours` от 0 до 24, неотрицательные шаги).

Если что-то не так — генерируется **`RequestValidationError`** (см. раздел про ошибки).

### Шаг 7. Что при невалидных данных

При ошибке валидации FastAPI **не доходит** до вашей функции `post_behavioral_analysis`. Вместо этого срабатывает обработчик **`RequestValidationError`** в `main.py`, и клиент получает **HTTP 422** с массивом `detail` (где именно ошибка).

### Шаг 8. Передача в service

Если тело валидно, в `post_behavioral_analysis` приходит уже готовый объект **`body: BehavioralAnalysisRequest`**.

Строка:

```python
return behavioral_service.analyze_behavioral(body)
```

передаёт этот объект в **сервисный слой** — функцию `analyze_behavioral` в `app/services/behavioral_service.py`.

### Шаг 9. Где считается `emotional_index`

В `analyze_behavioral` заводится переменная **`score = 100`**, затем применяются **штрафы** в зависимости от полей `payload` (сон, экран, шаги, общение). После этого результат ограничивается диапазоном **0..100** функцией **`clamp_int`** — это и есть **`emotional_index`**.

### Шаг 10. Как формируется `risk_level`

Функция **`_risk_level_from_index`** сопоставляет число:

- **0–39** → `"high"`;
- **40–69** → `"moderate"`;
- **70–100** → `"low"`.

### Шаг 11. Response model

Сервис возвращает **`BehavioralAnalysisResponse`**: Pydantic-модель с полями `emotional_index`, `risk_level`, `main_behavioral_signals`.

### Шаг 12. Из Python-объекта снова в JSON

FastAPI берёт возвращённый объект и **сериализует** его в JSON. Дополнительно, так как в декораторе указано `response_model=BehavioralAnalysisResponse`, ответ **фильтруется/приводится** к этой схеме (удобно для стабильного OpenAPI).

### Шаг 13. Ответ клиенту

Клиент получает HTTP **200** и JSON, например:

```json
{
  "emotional_index": 45,
  "risk_level": "moderate",
  "main_behavioral_signals": [
    "low_sleep",
    "high_screen_time",
    "low_physical_activity",
    "low_communication"
  ]
}
```

Этот JSON основной backend может сохранить в БД, отдать мобильному клиенту или объединить с другими модулями анализа.

## 4. Подробный разбор `main.py`

Ниже — **смысл блоков**, а не дословная строчная расшифровка каждой строки, но опираемся на реальный файл проекта.

### Импорты

Импортируются:

- **`FastAPI`** — фабрика приложения;
- **`HTTPException`, `Request`, `status`** — стандартные типы для HTTP и кодов ответов;
- **`http_exception_handler`** — встроенный обработчик HTTP-исключений Starlette/FastAPI;
- **`RequestValidationError`** — ошибка валидации тела/параметров;
- **`JSONResponse`** — явная JSON-ответа руками;
- роутеры из `app.api.v1.*`;
- **`get_settings()`** — настройки из `core/config.py`.

### `app = FastAPI(...)`

**Зачем:** создать **экземпляр приложения**. Именно его запускает uvicorn: `uvicorn app.main:app`.

**Параметры** `title`, `version`, `debug` попадают в OpenAPI. `docs_url` и `redoc_url` включают интерактивную документацию.

**Что такое `app`:** центральный объект, к которому **регистрируются маршруты** и **обработчики исключений**.

### `include_router(...)`

**Зачем:** подключить куски API из отдельных файлов. Каждый `APIRouter` знает только **свою относительную часть пути** (например, `/behavioral`), а **`prefix`** добавляет общий префикс (`/api/v1/analysis`).

**Как складывается путь:** `prefix` + путь в декораторе роутера.

У вас четыре роутера под одним префиксом — это нормально: позже в `conversation_emotion.py` и др. появятся свои `@router.post(...)`, и они не будут пересекаться по путям.

### Обработчик `RequestValidationError` → 422

Когда входные данные не проходят Pydantic, FastAPI бросает `RequestValidationError`. Ваш обработчик возвращает JSON:

```json
{ "detail": [ /* список ошибок с loc, msg, type */ ] }
```

со статусом **422 Unprocessable Entity**. Это стандартный выбор для «данные поняли, но они не проходят правила».

### Обработчик широкого `Exception`

**Зачем:** поймать **неожиданные** ошибки (баги, сбои библиотек), залогировать их на сервере и вернуть клиенту **безопасное** сообщение.

**Важный момент в вашем коде:** сначала проверяется `isinstance(exc, HTTPException)`. Если это **намеренный** HTTP-ответ (404, 401 и т.д.), он **делегируется** стандартному `http_exception_handler`, чтобы не превратить его в 500.

**Почему 500:** «внутренняя ошибка сервера» — когда что-то пошло не так **вне** штатной валидации.

### `/health` и `/`

**`/health`** — простой ответ `{"status": "ok"}`. Нужен **оркестраторам** (Kubernetes, Docker healthcheck): они периодически дергают URL и убеждаются, что процесс жив.

**`/`** — небольшая справка: имя сервиса, версия, ссылка на `/docs`. Удобно человеку и скриптам проверить, что поднялось именно это приложение.

## 5. Подробный разбор слоёв (API, schemas, services, utils, core)

### API layer (роутеры)

**Что делает router:** группирует endpoints и общие настройки (теги для OpenAPI, общий prefix при подключении).

**Почему не бизнес-логика:** роутер должен быть **тонким**: принял валидные данные → вызвал сервис → вернул результат. Так проще читать код и писать тесты: тест сервиса не требует HTTP.

**Зачем `@router.post`:** явно говорит, что ресурс создаётся/обрабатывается телом POST-запроса (для аналитики это типично).

### Schema layer (Pydantic)

**Request/response schemas:** это **формальное описание JSON**. Request — «что клиент обязан прислать»; response — «что сервер обещает вернуть».

**Зачем типы:** редактор подсвечивает ошибки, Pydantic ловит их в рантайме, FastAPI строит документацию.

**Вложенные модели:** пример — `DailyAnalysisRequest` в `schemas/daily.py`: поля `behavioral_result` и `conversation_result` имеют типы **других response-моделей**. Так вы говорите: «дневной модуль на входе получает уже **готовые результаты** двух подсистем», а не «плоский JSON без структуры».

**Контракт:** фронт, мобильное приложение и основной backend могут работать **по одному JSON-формату**, не зная внутренностей Python.

### Service layer

**Почему логика здесь:** это **ядро предметной области** — правила scoring сегодня, модели завтра.

**Замена baseline на ML:** вы меняете код внутри `analyze_behavioral` (или добавляете стратегии), сохраняя сигнатуру «на входе request-модель, на выходе response-модель». Роутер и URL остаются прежними.

### Utils layer

**Пример:** `clamp_int` в `helpers.py` — маленькая математика «не выйти за 0..100». Её могут переиспользовать другие сервисы, когда появятся новые индексы.

### Core / config

**Зачем:** префикс `/api/v1` и имя приложения **не зашиты** в каждом роутере — их можно поменять через переменные окружения (`EMOSENSE_API_V1_PREFIX` и т.д., см. `Settings` в `core/config.py`). Это ближе к **production**: разные окружения (dev/stage/prod) без правки кода.

## 6. Подробно про behavioral endpoint

### Request schema (`schemas/behavioral.py`)

Поля:

| Поле | Смысл |
|------|--------|
| `user_id` | Идентификатор пользователя (строка, не пустая) |
| `screen_time_hours` | Часы экрана в сутки, не меньше 0 |
| `sleep_hours` | Часы сна, от 0 до 24 |
| `physical_activity_steps` | Шаги, целое неотрицательное |
| `communication_frequency` | Частота/интенсивность общения, неотрицательное целое |

### Response schema

| Поле | Смысл |
|------|--------|
| `emotional_index` | Итог 0..100 |
| `risk_level` | `"high"` / `"moderate"` / `"low"` |
| `main_behavioral_signals` | Список коротких кодов причин (`low_sleep`, …) |

### Router (`api/v1/behavioral.py`)

Связывает путь `/behavioral` (после префикса) с функцией и указывает `response_model` для OpenAPI и стабильного ответа.

### Service (`services/behavioral_service.py`)

#### Откуда берётся 100

**100** — условный «максимум благополучия» перед штрафами. Это не «научная константа», а **удобная шкала для MVP**: чем больше нарушений образа жизни, тем ниже итог.

#### Почему штрафы

Каждый штраф кодирует **простую гипотезу**: мало сна, много экрана, мало движения, мало общения — хуже для эмоционального фона (в учебной модели).

#### Конкретные правила (как в коде)

- Сон **< 6** часов: **−20**
- Сон **< 4** часов: дополнительно **−10** (сильнее наказание за очень мало сна)
- Экран **> 8** часов: **−15**
- Экран **> 10** часов: дополнительно **−10**
- Шаги **< 5000**: **−10**
- Общение **< 3**: **−10**

#### `clamp`

После суммирования `score` может уйти ниже 0 или выше 100 в теории; **`clamp_int(score, 0, 100)`** приводит значение к допустимому диапазону **перед** вычислением `risk_level` и ответом.

#### `risk_level`

Диапазоны:

- **0–39** → высокий риск;
- **40–69** → умеренный;
- **70–100** → низкий.

#### `main_behavioral_signals`

Список **читаемых машиной** тегов, которые объясняют, **какие условия сработали** (не суммы, а флаги условий). Это удобно для UI, логов и объяснимости («почему индекс низкий»).

## 7. Ошибки и валидация

### Не пришло обязательное поле

Пример тела:

```json
{ "user_id": "x", "screen_time_hours": 1 }
```

Не хватает `sleep_hours`, `physical_activity_steps`, `communication_frequency`. Ответ **422**, в `detail` будет несколько объектов с `loc` вроде `["body", "sleep_hours"]`.

### Вместо числа пришла строка

```json
{
  "user_id": "x",
  "screen_time_hours": "много",
  "sleep_hours": 7,
  "physical_activity_steps": 8000,
  "communication_frequency": 5
}
```

Pydantic не сможет привести `"много"` к числу — снова **422** с понятным сообщением в `detail`.

### Почему именно 422

**422** означает: **синтаксис запроса в целом ОК**, JSON распарсился, но **семантика/типы/ограничения** не выполнены. Это отличается от **400 Bad Request**, когда JSON битый (не закрыта скобка) — тогда часто другая ошибка парсинга.

### Чем 422 отличается от 500

| Код | Когда | Кто «виноват» с точки зрения клиента |
|-----|--------|--------------------------------------|
| **422** | Данные не прошли правила | Клиент прислал неверный payload |
| **500** | Необработанное исключение на сервере | Обычно баг или сбой зависимости на сервере |

### Зачем глобальный обработчик исключений

Чтобы **не отдавать traceback** наружу и **логировать** проблему сервером (`logger.exception`). Клиент видит нейтральное `"Internal server error"`, а разработчик — полный лог на машине с сервисом.

## 8. Как это связано с основным backend

Упрощённая интеграция:

```text
Mobile App  ──HTTPS──▶  Main Backend (EmoSense)
                           │
                           │  HTTP POST JSON
                           ▼
                    ML API (этот проект)
                           │
                           │  JSON ответ
                           ▼
                    Main Backend
                           │
                           ├── сохранение в БД
                           ├── агрегация с другими данными
                           └── ответ в приложение
```

**Main backend** остаётся **единой точкой** для авторизации, бизнес-правил продукта и хранения. **ML API** — специализированный «калькулятор/инференс» с чётким контрактом.

Типичный сценарий:

1. Backend собрал сырые метрики (сон, шаги) из приложения.
2. Backend отправляет их в ML API на `/api/v1/analysis/behavioral`.
3. ML API валидирует и возвращает `emotional_index` и сигналы.
4. Backend решает, как интерпретировать это в продукте (уведомления, экран «заботы о себе», отчёт для куратора — зависит от политики проекта).

## 9. Почему это MVP и почему это нормально

### Почему rule-based baseline

На старте важно **запустить сквозной путь**: HTTP → валидация → логика → ответ. Простые правила:

- дают **объяснимый** результат (преподавателю и студенту проще защитить и понять);
- не требуют датасета и обучения на первом этапе;
- позволяют тестировать интеграцию с основным backend.

### Почему не нужно сразу LSTM/GRU

Нейросети требуют **данных, разметки, валидации, инфраструктуры**, иначе получится «чёрный ящик» с сомнительным качеством. MVP на правилах — способ **проверить архитектуру** и собрать реальные данные для будущих моделей.

### Как потом заменить реализацию

Сохраняете **схемы** и **URL**, меняете код в `services/behavioral_service.py` (или выносите модель в отдельный модуль и вызываете из сервиса). Роутер `behavioral.py` при этом может остаться **буквально тем же**.

## 11. Схема потока данных (ASCII)

```text
Клиент (Postman / Swagger / Main Backend)
        │
        │  HTTP POST, JSON в теле
        ▼
┌───────────────────┐
│  Uvicorn (ASGI)   │
└─────────┬─────────┘
          ▼
┌───────────────────┐
│  app: FastAPI     │  main.py — точка входа, обработчики ошибок
└─────────┬─────────┘
          │  маршрут POST /api/v1/analysis/behavioral
          ▼
┌───────────────────┐
│  APIRouter        │  api/v1/behavioral.py — выбор handler-функции
└─────────┬─────────┘
          │
          │  разбор и валидация тела
          ▼
┌───────────────────┐
│  Pydantic         │  schemas/behavioral.py — BehavioralAnalysisRequest
│  Request Schema   │  при ошибке → 422 (RequestValidationError)
└─────────┬─────────┘
          │  валидный объект body
          ▼
┌───────────────────┐
│  Service          │  services/behavioral_service.py
│  (бизнес-логика)  │  счёт score, clamp, risk, signals
└─────────┬─────────┘
          │  BehavioralAnalysisResponse
          ▼
┌───────────────────┐
│  Pydantic         │  та же схема как контракт ответа
│  Response Schema  │  + response_model в декораторе
└─────────┬─────────┘
          │  сериализация
          ▼
┌───────────────────┐
│  JSON HTTP 200    │  ответ клиенту
└───────────────────┘
```

**Расшифровка этапов:**

1. **Клиент** формирует HTTP-запрос.
2. **Uvicorn** принимает соединение и передаёт в приложение.
3. **`app`** находит нужный route по методу и пути.
4. **Router** вызывает вашу функцию endpoint.
5. **Pydantic** проверяет JSON → объект запроса.
6. **Service** считает результат предметной области.
7. **Response schema** гарантирует форму ответа.
8. **JSON** уходит обратно по сети.

## 12. Как читать этот проект новичку

Рекомендуемый маршрут **первого дня**:

1. **`app/main.py`** — как собирается приложение, какие префиксы и ошибки.
2. **`app/api/v1/behavioral.py`** — какой URL и куда уходит вызов.
3. **`app/schemas/behavioral.py`** — как выглядит «контракт» входа и выхода.
4. **`app/services/behavioral_service.py`** — где на самом деле считается индекс.
5. **`app/core/config.py`** — откуда берётся `/api/v1`.
6. **`app/utils/helpers.py`** — мелочь вроде `clamp`, которая участвует в результате.

**Почему такой порядок:** вы идёте **по потоку данных сверху вниз** — от HTTP к логике. Если начать с сервиса без роутера, непонятно, **кто** вызывает функцию. Если начать со схем без main — непонятно, **какой путь** у запроса.

После behavioral имеет смысл открыть **`schemas/conversation.py`**, **`daily.py`**, **`deep.py`** и заглянуть в **`services/conversation_service.py`** и др., чтобы увидеть **заготовки** следующих этапов.
